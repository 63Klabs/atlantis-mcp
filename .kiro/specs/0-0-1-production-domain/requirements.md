# Requirements Document

## Introduction

This feature introduces a centralized settings system for the postdeploy static site generation pipeline. A `settings.json` file provides stage-specific configuration values (such as footer HTML and production domain) that are applied to generated static content via token replacement. This replaces hardcoded footer HTML across all generated pages and enables API Gateway URLs to be rewritten to custom production domains.

## Glossary

- **Settings_File**: The JSON configuration file located at `application-infrastructure/src/static/settings.json` containing default and stage-specific key-value pairs.
- **Stage_Id**: The deployment stage identifier (e.g., `test`, `beta`, `prod`) provided by the `STAGE_ID` environment variable in the postdeploy pipeline.
- **Settings_Loader**: The component within the postdeploy pipeline responsible for reading the Settings_File, merging default values with Stage_Id-specific overrides, and producing a resolved set of key-value pairs.
- **Token**: A placeholder string in the format `{{{settings.<key>}}}` embedded in generated HTML or JSON content, intended to be replaced with the resolved value for that key.
- **Token_Replacer**: The component within the postdeploy pipeline responsible for finding all Token occurrences and the API Gateway URL pattern in generated files and replacing them with resolved settings values.
- **Api_Gateway_Url**: The URL pattern `https://{restApiId}.execute-api.{region}.amazonaws.com/{stageName}` used by API Gateway for the deployed REST API.
- **Postdeploy_Pipeline**: The sequence of shell scripts and Node.js scripts executed by `buildspec-postdeploy.yml` after CloudFormation deployment, responsible for generating and deploying static site content.
- **Consolidated_Output**: The `build/final/` directory containing all merged static content before S3 deployment.
- **Markdown_Docs**: The HTML files generated by Pandoc from markdown source files in script `03-generate-markdown-docs.sh`.
- **Api_Docs**: The Redoc HTML page generated by `resolve-and-render-spec.js` in script `02-generate-api-docs.sh`.
- **Openapi_Spec**: The exported OpenAPI 3.0 JSON specification file used for API documentation and available for download.

## Requirements

### Requirement 1: Settings File Structure

**User Story:** As a maintainer, I want a centralized settings file with default and stage-specific values, so that I can configure deployment-specific content from a single location.

#### Acceptance Criteria

1. THE Settings_File SHALL contain a `default` key whose value is an object of key-value pairs applied to all stages.
2. THE Settings_File SHALL support additional top-level keys matching Stage_Id values (e.g., `beta`, `prod`), each containing stage-specific key-value overrides.
3. THE Settings_File SHALL be valid JSON conforming to the structure `{ "default": { ... }, "<stageId>": { ... } }`.
4. WHEN the Settings_File contains keys `footer` and `domain`, THE Settings_File SHALL store their values as strings.

### Requirement 2: Settings Loading and Merging

**User Story:** As a maintainer, I want settings resolved by merging defaults with stage-specific overrides, so that each deployment stage receives the correct configuration.

#### Acceptance Criteria

1. WHEN the Postdeploy_Pipeline executes, THE Settings_Loader SHALL read the Settings_File from `application-infrastructure/src/static/settings.json`.
2. WHEN a key exists in both the `default` object and the Stage_Id-specific object, THE Settings_Loader SHALL use the Stage_Id-specific value, overriding the default.
3. WHEN a key exists in the `default` object but not in the Stage_Id-specific object, THE Settings_Loader SHALL use the default value.
4. WHEN a key exists in the Stage_Id-specific object but not in the `default` object, THE Settings_Loader SHALL include that key in the resolved settings.
5. IF the Settings_File does not exist or is not valid JSON, THEN THE Settings_Loader SHALL log an error and exit with a non-zero status code.
6. IF the Stage_Id does not match any top-level key in the Settings_File (other than `default`), THEN THE Settings_Loader SHALL use only the `default` values.

### Requirement 3: Footer Token Replacement

**User Story:** As a maintainer, I want the hardcoded footer HTML replaced by a settings-driven token, so that footer content can be managed centrally and vary by stage.

#### Acceptance Criteria

1. THE Markdown_Docs generation script (03) SHALL inject the Token `{{{settings.footer}}}` instead of hardcoded footer HTML into generated pages.
2. THE Api_Docs generation script (resolve-and-render-spec.js) SHALL embed the Token `{{{settings.footer}}}` instead of hardcoded footer HTML in the generated API documentation page.
3. WHEN the `footer` key is present in the resolved settings, THE Token_Replacer SHALL replace all occurrences of `{{{settings.footer}}}` in Consolidated_Output files with the resolved footer value.
4. THE static landing pages in `application-infrastructure/src/static/public/` SHALL use the Token `{{{settings.footer}}}` instead of hardcoded footer HTML.

### Requirement 4: Domain Token Replacement

**User Story:** As a maintainer, I want API Gateway URLs replaced with a custom production domain in generated documentation, so that end users see the correct domain for their environment.

#### Acceptance Criteria

1. WHEN the `domain` key is present in the resolved settings, THE Token_Replacer SHALL replace all occurrences of `{{{settings.domain}}}` in Consolidated_Output HTML and JSON files with the resolved domain value.
2. WHEN the `domain` key is present in the resolved settings, THE Token_Replacer SHALL replace all occurrences of the Api_Gateway_Url pattern in Markdown_Docs files with the resolved domain value.
3. WHEN the `domain` key is present in the resolved settings, THE Token_Replacer SHALL replace all occurrences of the Api_Gateway_Url pattern in the Api_Docs HTML file with the resolved domain value.
4. WHEN the `domain` key is present in the resolved settings, THE Token_Replacer SHALL replace all occurrences of the Api_Gateway_Url pattern in the Openapi_Spec JSON file with the resolved domain value.
5. IF the `domain` key is not present in the resolved settings, THEN THE Token_Replacer SHALL skip domain replacement and leave Api_Gateway_Url patterns unchanged.

### Requirement 5: Generic Token Replacement

**User Story:** As a maintainer, I want all `{{{settings.<key>}}}` tokens replaced with their resolved values, so that future settings keys are automatically supported without code changes.

#### Acceptance Criteria

1. FOR ALL keys in the resolved settings, THE Token_Replacer SHALL replace occurrences of `{{{settings.<key>}}}` in all files within the Consolidated_Output directory with the corresponding resolved value.
2. IF a Token references a key that does not exist in the resolved settings, THEN THE Token_Replacer SHALL leave that Token unchanged in the output file.
3. THE Token_Replacer SHALL process HTML files and JSON files within the Consolidated_Output directory.

### Requirement 6: Pipeline Integration

**User Story:** As a maintainer, I want the settings loading and token replacement integrated into the existing postdeploy pipeline, so that the process runs automatically during deployment.

#### Acceptance Criteria

1. THE Token_Replacer SHALL execute after all content has been copied into the Consolidated_Output directory and before the S3 sync in script `04-consolidate-and-deploy.sh`.
2. WHEN the Token_Replacer executes, THE Postdeploy_Pipeline SHALL log the number of replacements performed for each settings key.
3. THE Postdeploy_Pipeline SHALL use the `STAGE_ID` environment variable to determine which stage-specific settings to apply.
4. IF the Token_Replacer encounters an error during file processing, THEN THE Postdeploy_Pipeline SHALL log the error and exit with a non-zero status code.

### Requirement 7: Downloadable OpenAPI Spec with Domain Replacement

**User Story:** As a maintainer, I want the downloadable OpenAPI spec to contain the custom domain instead of the API Gateway URL, so that end users can import the spec with the correct server URL.

#### Acceptance Criteria

1. THE Postdeploy_Pipeline SHALL copy the exported Openapi_Spec JSON file into the Consolidated_Output directory so it is available for download.
2. WHEN the `domain` key is present in the resolved settings, THE Token_Replacer SHALL replace the Api_Gateway_Url in the `servers[].url` field of the Openapi_Spec JSON with the resolved domain value.
